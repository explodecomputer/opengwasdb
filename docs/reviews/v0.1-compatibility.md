# v0.1 Dense Observed-Only Compatibility Review

Status: implementation review note

This review checks the first vertical slice against `docs/spec/store-format.md`
and ADRs 0001, 0005, 0006, 0007, 0008, 0011, 0013, and 0015.

## Findings

- The public query entry point is `StoreQuery`, selected from the opened Store
  Release manifest. Result rows are layout-independent association records and
  do not expose Dense row/column storage details as the primary user contract.
- Source ingestion and canonical variant normalisation live in
  `opengwasdb.build.source`; Dense writing lives in `opengwasdb.layouts.dense`.
  This keeps the path open for Ragged writers to reuse normalised records.
- The manifest records layout, association coverage, completion state, reference
  assembly, build provenance, statistic arrays, dense dtype, chunk shape,
  compressor, and top-hit thresholds. Reference Completion-specific metadata is
  not required for Observed-Only releases, but the manifest model accepts
  provenance objects large enough to record the `pleiodb` method, LD block TSVs,
  `.unphased.vcor1.gz` matrices, optional `.ldeig.rds` files, and cached
  eigenfactor extracts in a future Reference-Completed release.
- Query results currently omit Association Status and Query Component because
  v0.1 only implements Dense Observed-Only. The result dataclass can add those
  optional fields without changing the existing core fields: variant identity,
  analysis identity, Z, SE, derived beta, p-value, and stored effect scale.
- Missing Dense Observed-Only cells are represented by NaN in both `z` and `se`
  and are excluded from query result streams. This is compatible with the
  Reference-Completed NaN-plus-imputed-mask design in ADR 0013.

## Follow-up risks

- Zarr v3 has materially different codec metadata behaviour from Zarr v2. The
  v0.1 package pins `zarr<3` until the store-format spec deliberately adopts or
  tests Zarr v3 semantics.
- The v0.1 normaliser validates simple A/C/G/T alleles only. Left-alignment,
  trimming, and long-allele hashing are deferred and should be implemented
  before production ingestion of complex indels. Follow-up:
  `issues/012-production-variant-normalisation.md`.
